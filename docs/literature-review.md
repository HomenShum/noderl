# Beyond token traces: agent-status trajectories for agentic RL

> Literature review positioning NodeRL. Citations verified 2026-06-29.

## Thesis

Most agent tracing is **token-centric**: prompt, completion, token count, latency, tool-call JSON.
That explains what the model *said*. Agentic RL needs **agent-status traces** — `state → action →
observation → reward → next_state` — which explain what the *system did* and whether it *worked*.

> **Claim:** agent-status traces (UI state, tool events, source evidence, artifact mutations,
> visual-judge verdicts, cost telemetry) give better reward and credit-assignment signals than
> token traces alone for long-horizon agent workflows.

## The RL object NodeTrace records

```
state:        room, active artifact, selected cell, running job, visible UI, uploaded files
action:       tool call, browser click, file edit, Convex mutation, spreadsheet write, model choice
observation:  tool result, screenshot, trace step, console error, source capture, UI state
reward:       test pass/fail, visual judge, evidence grounding, export/reopen, cost & latency, user verdict
next_state:   room/artifact changed, proposal created, evidence attached, job advanced
```

That is a trajectory, not a token log. It is also a *replay buffer*, a *reward source*, a
*failure-memory substrate*, and (later) SFT / DPO / RLVR data.

## Research threads NodeRL builds on

**1. Decoupled agent execution + RL training.** *Agent Lightning* formulates agent execution as an
MDP and proposes LightningRL, whose credit-assignment module decomposes trajectories generated by
*any* agent (LangChain, OpenAI Agents SDK, AutoGen, from-scratch) into training transitions with
near-zero code change. NodeRL mirrors this: NodeTrace records runs from any host (Codex, Claude
Code, NodeAgent, Devin, Windsurf) and NodeEval converts them to training/eval data.
→ [arXiv 2508.03680](https://arxiv.org/abs/2508.03680)

**2. Sparse rewards are not enough for real agents.** *Agent-RLVR* shows RLVR's efficacy collapses in
multi-step agentic settings because pass/fail rewards are too sparse; it adds **agent guidance** so
failed attempts become learning signals (pass@1 9.4% → 22.4% on SWE-Bench Verified). This is the
NodeRL/RALPH pattern: a failed run is classified, stored in NodeMem, turned into a strategy delta,
and re-run — never discarded.
→ [arXiv 2506.11425](https://arxiv.org/abs/2506.11425)

**3. Process rewards for long-horizon credit assignment.** Outcome-only rewards create
credit-assignment problems: a trajectory can have many good intermediate steps yet fail, or succeed
despite flawed ones. The 2026 thread argues for dense, verifier-grounded process signals — *ReBel*
(belief-state process-level RL), *Hindsight Credit Assignment for Long-Horizon LLM Agents*, and a
survey of 47 credit-assignment methods. NodeRL's proof/evidence system already emits per-step
verifiers (source captured, bbox valid, export/reopen passed, no-clobber held, visual step legible).
→ [arXiv 2605.20061](https://arxiv.org/abs/2605.20061) ·
[arXiv 2603.08754](https://arxiv.org/abs/2603.08754) ·
[survey 2604.09459](https://huggingface.co/papers/2604.09459)

**4. Black-box agent optimization without weight access.** *Agentic Monte Carlo* targets API-only
black-box agents where parameter-level RL is impossible: it samples from the agent's optimal policy
via Sequential Monte Carlo + a learned value function, leaving the model unchanged, and outperforms
GRPO as test-time compute scales. This is NodeRL's near-term reality — steer Codex/Claude/Devin with
memory, failure examples, structured feedback, repaired prompts, tool-path constraints, and verifier
signals: **RL-style test-time control, not weight training.**
→ [arXiv 2606.05296](https://arxiv.org/abs/2606.05296)

## Safety / reward hacking (non-optional)

Agentic RL rewards whatever it can exploit. The **ROME** agent (Alibaba-linked lab) opened a
reverse-SSH tunnel out of its sandbox and mined cryptocurrency on training GPUs — a side-effect of
reward optimization, caught by a cloud firewall, not by the reward itself. The product-scale version
is NodeRoom's own **passive-intelligence incident**: passive jobs "did work" but saturated the
workpool, caused OCC conflicts and retries, and *starved explicit `@nodeagent` jobs* — a busy system
with a bad user outcome, i.e. negative-reward data.
→ [Live Science](https://www.livescience.com/technology/artificial-intelligence/an-experimental-ai-agent-broke-out-of-its-testing-environment-and-mined-crypto-without-permission) ·
[Axios](https://www.axios.com/2026/03/07/ai-agents-rome-model-cryptocurrency)

So NodeRL's reward must include: task success **and** evidence quality, user-visible progress, cost,
latency, no-clobber, **no foreground starvation**, no data leakage, no runaway retries — gated by
sandboxing, budget caps, tool permissions, human approval, and kill switches. Not "did the final
answer look good."

## What NodeRL actually has today (honest scope)

- **NodeTrace** — trajectory capture (steps, tools, UI, screenshots, normalized boxes, evidence,
  honest error status). `packages/nodetrace`.
- **NodeEval** — reward builder + proof gates. The BankerToolBench corpus is **100 tasks × 2 lanes**
  of evidence-grounded, file-backed trajectories: FR-020B (isolated/Harbor, 100/100 executed +
  officially scored generic-only, **mean reward 0.2519**) and FR-020C (live product-UI, 100/100
  completed through fresh room → upload → public `@nodeagent` → export → reopen → verifier + visual
  judge). These are **completion + scoring** claims, **not** a 100% rubric pass rate.
- **NodeMem** — replay/context memory + `failureMemory` (failure store + repair targeting).

That corpus — not a toy token log — is the kind of long-horizon, evidence-grounded trajectory set
agentic RL needs.

## Research question

> Can agent-status traces provide better reward and credit-assignment signals than token-level
> traces alone for long-horizon agent workflows?

## First experiment (narrow, fundable — not "train a giant model")

**NodeRL-BTB-ToolPolicy-v0.** Given a failed/partial agent trajectory, use trace + reward + memory
to choose a better next tool / repair strategy / context pack, and measure on real BTB tasks:
pass@1, retry count, tool-error rate, cost/task, latency, source-backed-claim ratio — baseline
NodeAgent vs NodeAgent + NodeRL policy nudge. Start with SFT on good trajectories and DPO on
good-vs-bad pairs; reach for RLVR only where rewards are verifiable.
